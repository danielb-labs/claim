claim

A tool that checks whether the facts you wrote down are still true.

Every codebase is full of recorded reasons: a security exception that says a vulnerability doesn’t apply, a dependency pinned to an old version, a skipped test, a line in an agent’s context file. Each was true when someone wrote it. Nothing checks it afterward, so it quietly rots — and the person who later makes it false usually has no idea the note exists.

claim binds each of those facts to a way of re-checking it. When a fact stops being true, the system notices and says so, instead of leaving a confident sentence that is now wrong. The one-line version: recorded knowledge either re-verifies itself or comes back to a human — the failure mode is a nag, never a lie.

The core idea

A claim is three things written down together:

  1. a statement — a fact, in plain language;
  2. a check — a way to re-verify the fact;
  3. a schedule — when the check should run.

You write claims where you already record decisions. The tool runs the checks on their schedule and keeps a history of what each one found. A passing check keeps the claim fresh; a failing check marks it drifted and puts it in front of whoever the fact matters to. That’s the whole loop: record → check → notice drift → adjudicate.

Why not just write a test? A failing test asks “is the change that triggered me wrong?” When the honest answer is “no, the world legitimately changed,” whoever tripped the test edits it away, and the reason it existed is lost. A claim carries the reason with it and routes drift to the decision’s owner, not to whoever happened to change the world. (Some facts should become tests — after they’ve earned it; that is what claim retire with a closing note is for.)

Getting started

Install

Build the CLI from source with Rust’s package manager, from a checkout of this repository:

cargo install --path crates/claim
claim --help

The binary lands on your PATH at ~/.cargo/bin/claim. To run against a development build instead, use cargo build -p claim and ./target/debug/claim.

A first claim

In a git repo, scaffold a store and record a claim about a pinned dependency:

# set the scene
echo 'libfoo==4.2' > requirements.txt
git add -A && git commit -m init

claim init                         # create the .claims/ store

claim add \
  --id libfoo-pin \
  --statement "Pin libfoo at 4.2 — 5.x corrupts CJK PDF export." \
  --run "grep -q 'libfoo==4.2' requirements.txt" \
  --max-age 120d \
  --supports requirements.txt#libfoo

The --run command is the check: it exits 0 while the pin holds. add runs it once, requires it to pass (a passing check is the verification), and writes the claim under .claims/ with its first recorded result. It never touches your working tree. Commit the new files like any other:

git add .claims && git commit -m "record libfoo pin claim"

Every command takes --json for machine-readable output; the flag may lead or follow the verb (claim --json list or claim list --json). At an interactive terminal, add prompts for any required field you omit; a non-interactive caller (an agent or CI) that omits one gets a clear error instead.

Claims

A claim lives in one Markdown file under .claims/. The body is the statement a human reads; the front-matter is what the tool acts on.

---
id: libfoo-pin
checks:
  - kind: cmd
    run: "grep -q 'libfoo==4.2' requirements.txt"
    when: on-change
max-age: 120d
supports: [requirements.txt#libfoo]
---
We pin libfoo at 4.2. Versions 5.x corrupt PDF export for CJK fonts.

The same schema can also travel inside another file (like an agent’s CLAUDE.md) as an <!-- claim ... --> HTML comment block, so a context file can carry the checks that keep it honest. (The CLI authors and reads standalone .claims/*.md files; embedded blocks are part of the format for hand-authoring.)

id
A required kebab-case slug, unique in the store, optionally namespaced with / (e.g. payments/libfoo-pin).
statement
The Markdown body: the fact, in plain language — what a person reads when the claim comes up for review. Required; a claim with no statement is rejected.
checks
A required, non-empty list. Each check has a kind (cmd, agent, or human) and a when trigger. See Checks & verdicts.
max-age
A required <N>d duration — the freshness window. See Status & freshness.
supports
An optional top-level list of the decisions this fact justifies — a decision ref like requirements.txt#libfoo or another claim’s id. See Supports & drift.

Notably absent: no author, reviewer, owner, or status field. Anything typed into a file can be forged or go stale, so those are derived, not stored — see The store & git.

Checks & verdicts

A check is how a claim gets re-verified. The most common kind is cmd, an ordinary command: the tool runs it and reads its exit code. (agent checks, for facts no command can settle, are covered below; human checks are a scheduled human look.) The tool turns a check’s outcome into a verdict. This mapping is the heart of the tool, so it is deliberately strict:

The cmd check…VerdictMeans
exits 0heldthe fact is still true
exits 1driftedthe fact is no longer true
any other exit, fails to spawn, is signalled, or times outbrokenthe check itself couldn’t run

The rule that matters: only a clean exit 0 counts as “still true.” A missing tool, a deleted directory, a typo, a hung command — anything that stops the check from completing — is broken, which ages the claim toward review exactly like a check that never ran. There is no path from “couldn’t check” to “it’s fine.” A fourth verdict, unverifiable (ran, but the evidence was inconclusive), exists for agent checks; a plain cmd check never returns it.

Negation

Some facts are phrased as an absence (“the archived serde_yaml is not in the lockfile”). Set negate: true and the tool flips held and drifted — but never turns broken into a pass. The tool owns the inversion; you never write a shell !, because a missing tool would then flip a failure into a false “fine.” On the CLI this is --negate; in the file it is the check’s negate field.

Triggers — when a check runs

Each check carries a when:

  • on-change — for facts only a code change can break (“nothing imports X”). Runs in the on-change lane, on every pull request.
  • every 30d — for facts the outside world can break (“the vendor’s rate limit is still 100/s”). Runs on a clock, since no commit of yours would signal the change.

These are the two triggers in v1. A claim is “due” when it has an on-change check (always, in v1) or when an every Nd check’s interval has elapsed since its last recorded run. claim check defaults to running only the due claims; --all runs every claim’s checks regardless.

Status & freshness

A verdict is the result of one check at one moment. A claim’s status is the standing you get by looking at its whole history. It is never stored — it’s computed from the verdict log and the claim’s max-age every time you ask.

StatusMeaning
verifiedthe latest check held and the claim is within its max-age
driftedthe claim’s own check says the fact is no longer true
staleoverdue — never verified, past its max-age, or its checks have been broken or inconclusive too long
retiredclosed on purpose — the decision was re-reviewed and the claim was put to rest with a note

(suspect — a claim whose foundation drifted — is in the design but not v1; until it ships, the drift report lists what a drifted claim supports, which at small scale is the same information.)

max-age — the dead-man’s switch

Every claim carries a required max-age. A passing check renews it. If checks stop passing — the check broke, kept coming back inconclusive, or there was never a check at all — the claim eventually goes stale and a human is asked to look. This is the backstop under everything: whatever else goes wrong, the end state is a nag, never a stale green light. A claim hand-committed with no recorded verdict is not a special state — it is simply stale, and due immediately, until a check runs.

Because staleness is arithmetic against a log, nothing has to run for a claim to become stale, the same way nothing runs for a certificate to expire. What has to run is something that asks regularly, so staleness gets noticed — that is the clock lane’s job, or, on a bare single machine, the next time anyone runs the CLI.

Witnessing the red

A check is only worth trusting if it can actually fail. It’s easy to write one that passes today but would never catch the fact breaking — a grep for the wrong string, a command that always exits 0. That check is green now and green forever, which is worse than no check, because it looks like protection.

Witnessing a check go red is an optional confidence signal, not a gate. A passing check already verifies the fact, so claim add records a claim the moment its check holds against reality. Where a fact’s failure can be staged, you can ask the tool to watch the check discriminate: pass --witness-cmd with a command that makes the fact false. The tool applies that command in a throwaway git worktree detached at HEAD, runs the check there expecting drifted, and tears the worktree down — so your working tree is never touched and no clean tree is required. The observed red is recorded as evidence on the establishing verdict. (It needs a commit to check out, so it is refused on a brand-new repo with no commit yet.)

It’s the same as testing a smoke alarm by holding a match under it — nice to see it go off before you trust it, but a fact whose red can’t be staged (a future world-event, not a file you can edit) is still verified by its passing check, never marked “unverified” for a red nobody could produce. claim list --unverified surfaces claims with no passing verdict on record at all — genuinely never verified, not merely un-witnessed.

Supports & drift

A claim can name the decisions it justifies, with supports. The “user input never reaches XmlParser” claim supports a security exception; a “PDF export uses CJK fonts” claim supports the libfoo pin. This is what lets a drift carry meaning: when a fact breaks, the tool can point at what rested on it.

A supports target is either a decision ref (path#anchor) or a bare claim id or file path. A decision ref resolves when path exists and the literal text anchor occurs in it as a case-sensitive, word-boundary text scan — not a GitHub heading slug. To point at a heading “## Approved dependencies”, write the words as they appear (CLAUDE.md#Approved dependencies), not #approved-dependencies. claim add checks each target and warns on any that does not resolve (a forward reference is allowed, so it is a warning, not a hard failure); claim check later reports an unresolved target as review-needed.

When a claim’s check confirms the fact is false, its status becomes drifted. Drift is not a build failure — it’s a review item. The right response is one of:

  • claim amend — the fact changed shape; update the statement and check, keeping the claim’s history.
  • claim retire — the decision was re-reviewed and no longer needs the fact (or the fact became a real test, and a closing --note says where).

Every check run also confirms a claim’s supports targets still resolve. If the decision a claim justified was deleted, the claim goes loud instead of sitting there quietly green — an unresolved support is exit 1, review-needed.

Agent checks (kind: agent)

Many facts have no command that settles them — “has the upstream CJK-corruption bug been fixed in libfoo 5.x?” is a reading of a changelog and an issue tracker, a judgement. An agent check binds that judgement to a natural-language instruction that an agent (a model behind a CLI) carries out, returning a verdict and the evidence for it.

Agent checks are strictly opt-in. They execute only when the CLAIM_AGENT_CMD environment variable points claim check at a runner — a shell command that receives the prompt on stdin and prints a verdict JSON object on stdout. With it unset (the default), every agent check is reported unverifiable, no subprocess is spawned, and no model is contacted — a plain claim check never makes an API call. The runner, its credentials, and its budget are the operator’s; the tool ships no model client.

The honesty contract is identical to a cmd check: a crash, a timeout, a non-zero exit, a missing or malformed verdict, or two conflicting verdicts all map to broken, never a fake held. A runner cannot fake a pass by claiming one — it must exit 0 and emit exactly one well-formed held on stdout. The full contract, response schema, and the exact verdict mapping are in docs/agent-checks.md.

The store & git

Claims are plain files in a .claims/ directory, and their history is an append-only log beside them (.claims/log/<id>/, one small file per result). There is no database and no server: git is the store. That means claims are versioned, reviewed, merged, and attributed exactly like code, and two people (or agents) recording claims at the same time is just two commits.

The store scanner treats every .md under .claims/ that opens with a --- frontmatter fence as a claim, and skips the log/ tree. A plain document with no fence — a README.md describing the store — is not a claim and is skipped silently, so it can coexist with claims. A file that does open with a fence but is malformed stays a loud per-file error rather than being dropped.

One consequence worth stating: a claim’s author, its reviewer, and its status are never written into the file, because anything typed into a file can be forged or go stale. Author and reviewer come from the commit and the pull request; status is computed from the log. The file holds only what someone decides — the statement, the check, the schedule.

The honesty rules

The tool’s whole value is that you can trust what it says, so a handful of golden invariants are non-negotiable (they live in CLAUDE.md):

  • A broken check never reports a pass. Exit 0 → held; exit 1 → drifted; any other exit, a failure to spawn, a signal, or a timeout → broken. Broken counts against freshness exactly like “never checked.”
  • The tool owns negation. Inversion happens in the tool, never as a shell ! that a missing binary could flip into a false pass.
  • Status and provenance are derived, never stored. Status comes from the log; author and reviewer from git and the forge — not from fields a file could lie about.
  • A write to the truth is a commit. The tool appends verdicts as files to commit; there is no side channel, no database, no write API to the truth.
  • Witnessing a red is optional confidence, never a lie. A passing check verifies the fact; a witnessed red adds confidence where a failure can be staged.
  • The failure mode is a nag, never a lie. Every path degrades toward a human being asked to look, never toward a stale green light.

Lifecycle

A claim exists once its file is merged into a store. From there its status moves with what its checks find and what a human decides:

Architecture

The tool is built in layers, each usable without the ones above it: plain claim files, git as the history and review layer, and (in the design, not v1) an optional hub that indexes many repos without ever becoming the source of truth.

Doubt travels along supports edges, never confidence: when a claim drifts, everything resting on it is flagged for a second look, but a verified parent never makes a child true — only the child’s own check can. (Propagation and the suspect status are in the design; v1 shows the one-hop supports listing instead.)

CI: the two lanes

claim verifies facts in two places, on two clocks. Both end at the same verdict log and neither ever fails your build — drift is information routed to an owner, not a wall in front of a merge.

  • The on-change lane runs on every pull request (claim check --all --report-only --json). It answers “does this diff break a fact someone wrote down?” and posts one grouped, advisory comment on drift — the statement, what it supports, and the CODEOWNERS owner. It writes nothing and needs no write token, so a fork PR still gets its checks.
  • The clock lane runs on a schedule (claim check --due --json), records each due claim’s verdict, and commits the log files back (a write to the truth is a commit). It maintains one standing claim: due & drifted issue — created when there is a queue, updated in place, closed when the store is clean. That issue is the entire v1 nag mechanism.

The renderer that turns claim check --json into a comment or issue body lives in ci/render.mjs (Node, no dependencies) and is tested against real fixtures, so a change to the CLI’s JSON shape breaks the build rather than mis-rendering in production. Both lanes and their exit-code contract are documented in full in docs/ci.md; drop-in workflow copies are in examples/consumer/.

The exit-code contract (highest applicable wins): 0 — every check held and every support resolved; 1 — a drift, an unverifiable, or an unresolved support (review needed); 2 — a broken check, an unloadable or duplicate-id claim file, or a tool error. A broken check is the loudest condition, never the quietest.

The MCP server

The claim-mcp server is how agents touch the store over the Model Context Protocol. It is a thin shell over the same core the CLI uses, discovering the store from the working directory per call, and exposes three tools:

  • query — return the claims recorded for the paths or text at hand, as dated evidence: the statement, its computed status, when it was last verified, what it supports, and an evidence pointer. Run it at the start of work to inherit what the store already believes. Results are framed as observations to weigh against their freshness, never as instructions to obey — a claims store agents obeyed blindly would be an injection channel with a trust stamp. Read-only; it never writes.
  • report — record a verdict the agent reached (held, drifted, or unverifiable) with required evidence about an existing claim, under the agent’s own git identity. It appends one entry to the verdict log in the working tree and returns the file to commit — the server does not commit, because a write to the truth is a commit the caller makes.
  • create — record a new claim the agent just established: a statement plus the check that re-verifies it, verified against reality now, under the agent’s own git identity. This is the MCP counterpart of claim add, held to the same rule — the check is run against the current tree and the claim is written only if it held; a drifted (the fact is already false) or broken/unverifiable (the check could not run) is refused with the evidence and nothing is written. Like report, it writes the claim file and its establishing verdict to the working tree with a commit hint and does not commit, so the new claim is unreviewed until the caller commits it (where a human reviews the commit or PR). An instruction (agent) check needs a runner (CLAIM_AGENT_CMD); with none it is unverifiable and the create is refused.

(propose and flag from earlier drafts were cut: propose’s role — an agent authoring a claim — is filled by create writing a file the caller commits as a reviewable PR, and flag is just report with a negative verdict.)

CLI reference

Every command takes --json for machine-readable output (it may lead or follow the verb). All verbs below are shipped. Run claim <verb> --help for the authoritative, always-current flag list; the summaries here match version 0.1.0.

CommandDoes

claim init

Scaffolds .claims/ and .claims/log/ in the current directory (or --dir <DIR>). Idempotent: re-running is not an error.

claim add

Records a claim after running its check once and requiring it holds — a drifted or broken check is refused (no recording an already-false or unrunnable fact). Flags: --id, --statement, --run, and --max-age are required (prompted at a terminal, a clear error otherwise); --when defaults to on-change; --negate inverts the check’s sense; --supports <TARGET> is repeatable; --witness-cmd optionally witnesses a red (see above).

claim check

Runs checks and records the verdicts, scriptable by exit code. By default it runs only due claims; --due names that explicitly and --all runs every claim’s checks (mutually exclusive). --report-only runs and reports but writes nothing — the untrusted-runner mode for a fork PR or a local sanity pass; the exit code is still set from the verdicts, but nothing is persisted (so an every Nd claim stays due until a persisting run records it). Retired claims are never checked. Set CLAIM_AGENT_CMD to execute kind: agent checks; unset, they are unverifiable and nothing is spawned.

claim list

Inventories claims with their computed status. Optional [TERM] is a case-sensitive substring matched against id and statement. Filters: --status <verified|drifted|stale|retired>, --path <PREFIX>, --supports <TARGET>, --stale (shortcut for --status stale), and --unverified (claims with no passing verdict on record).

claim log

claim log <ID> prints one claim’s definition and its full verdict history — each entry’s timestamp, verdict, the commit it was taken against, the git-derived actor, and any evidence (including a witnessed-red note on an establishing entry).

claim drift

Lists only the drifted claims, each with what it supports — the review queue. Exit 0 when nothing drifted, 1 when the queue is non-empty, 2 on a tool error.

claim amend

claim amend <ID> replaces parts of a claim while keeping its history: --statement, --run (with --negate, which is only meaningful together with --run), --when, --max-age, and --supports (repeatable; replaces the whole set — it never silently drops edges it was not told to).

claim retire

claim retire <ID> --note <WHY> closes a claim on purpose; the required note records why (the decision was re-reviewed, or the fact became a real test and this says where). Its status becomes retired and it is no longer checked.

claim stats

Pilot metrics: claim counts by status, verdict counts across all history, drifts caught, and the oldest last-verified time. The two “kill metrics” from the product design — false-alarm rate and minutes per claim — are reported as needing human input rather than fabricated, because the store does not observe them.

claim docs

Opens this documentation site — the very page you are reading — from a copy bundled into the binary, so an installed user with no checkout of this repository can still read the docs, and the docs they read are version-locked to the tool they ran (an old binary never shows a newer site than it was built from). It writes the site to a per-version cache directory and hands it to the platform opener (open, xdg-open, or start), printing the path too. claim docs --path prints the path without opening — for a headless box or scripting (open "$(claim docs --path)"); a box with no opener degrades to the same, printing the path and exiting 0 rather than failing. claim docs <topic> opens a deeper page: ci, agent-checks, or dogfooding.

FAQ

Is this meant for AI agents or people?

Both. People read claims to understand what the team believes and why; agents read them at the start of a session (through the MCP query tool or --json) to inherit verified premises instead of re-deriving the world, and record what they learn with report. The interactive prompts you may hit only appear for a person at a terminal — run non-interactively with flags and the tool never prompts.

Is a drifted claim a build failure?

No. Drift is a review item routed to the decision’s owner, not a red build for whoever changed the world. Neither CI lane fails the build; the escalation ladder in v1 has exactly two rungs — the on-change PR comment and the clock-lane standing issue — and neither blocks a merge.

Do I have to witness a check fail before recording it?

No — that used to be mandatory and is now optional. A passing check verifies the fact, so claim add records it the moment the check holds. --witness-cmd is there when you want the extra confidence of seeing the check go red, and it runs in an isolated worktree that never touches your files.

What happens if I never run the checks?

Claims still age. Because max-age is measured against the log, a claim nobody checks eventually reads as stale — the tool would rather nag you than quietly imply a fact still holds.

Do I have to write claims for everything?

No — the opposite. A claim is only worth it where you were already recording a decision whose justification cites a fact someone else could change. If nobody would have written the sentence anyway, don’t make it a claim.
